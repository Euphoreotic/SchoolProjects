Assignment 4 - Inheritance

Due by onQ Submission Before 7pm, Monday Nov. 12

  Assignment Statement

The Fastenal web site provides a very nice illustration of a natural hierarchy. You will use this representation to help you build a code model of a small portion of this hierarchy for Fasteners. Your portion of the hierarchy will need to hold descriptions of Carriage Bolts, Wood Screws, Wing Nuts and Common Nails only:

A Carriage Bolt is described by the following fields:

  • Diameter / Thread Size
  • Length
  • Material
  • Finish
  • Unit Price
  • Number per Unit
These fields would describe any Bolt, but CarriageBolt should extend Bolt.

A Wood Screw is described by the following fields:

  • Diameter / Thread Size
  • Length
  • Material
  • Finish
  • Head
  • Drive
  • Point
  • Unit Price
  • Number per Unit
Point is unique to Wood Screws, but other Screws would contain all other fields.

A Wing Nut is described by the following fields:

  • Diameter / Thread Size
  • Material
  • Finish
  • Unit Price
  • Number per Unit

A Common Nail, or any Nail for that matter, is described by the following fields:

  • Material
  • Finish
  • Size
  • Length
  • Gauge
  • Unit Price
  • Number per Unit

Not all fields have been included for these fasteners - Manufacturer, Grade, Standard and Thread Type have been left out.

  Restrictions on Field Values

Each of the fields listed above are restricted to certain values. The inches symbol, ", can be omitted from the attribute values. Some further simplifications have been applied to those shown on the Fastenal site.

  • Materials for all Fasteners: Brass, Stainless Steel, Steel.
  • Finishes for Steel only: Chrome, Hot Dipped Galvanized, Plain, Yellow Zinc, Zinc.
  • Finish for Brass and Stainless Steel Materials: Plain.
  • Additional Finishes for Steel Screws only: Black Phosphate, ACQ 1000 Hour, Lubricated.
  • Diameter/Thread Size for any threaded Fastener: #8-13, #8-15, #8-32, #10-13, #10-24, #10-32, 1/4"-20, 5/16"-18, 3/8"-16, 7/16"-14, 1/2"-13, 5/8"-11, 3/4"-10.
  • Lengths for Bolts and Screws: 1/2" to 6" in units of 1/4", 6" to 11" in units of 1/2", 11 to 20" in units of 1".
  • Head for all Screws: Bugle, Flat, Oval, Pan, Round.
  • Drive for all Screws: 6-Lobe, Philips, Slotted, Square
  • Point for Wood Screws: Double Cut, Sharp, Type 17
  • Material for all Nails: Steel.
  • Finishes for Common Nails: Bright, Hot Dipped Galvanized.
  • Sizes for Common Nails: 6D, 8D, 10D, 12D, 16D, 60D.
  • Lengths for Common Nails: 2, 2.5, 3, 3.25, 3.5, 6 (in inches).
  • Gauge for Common Nails: 2, 8, 9, 10.25, 11.5

"Number per Unit" must be at least 1, but could be as high as 10,000. This number will always be one or if higher, evenly divisible by 5. The Unit Price, in dollars, must be greater than zero and is the price per unit, where a unit could be anywhere from 1 to 10,000 fasteners.

  Details

Here are some more instructions and restrictions on what you must do:

  • Use this JUnit, version 5.1, testing class to put your hierarchy through its paces: TestHierarchy.java. You may add tests to this class, but cannot remove any.
  • After passing all unit tests, run the "order system" class: FastenerOrderSystem.java. You cannot change the code in this class, and it will provide more information on how your classes must behave.
  • Try to allow your hierarchy to be extensible. In other words you should try not to build any structure that would prevent the addition of other types of Fastener objects without requiring modifications to the existing structure. Once you have built your hierarchy, pick a few Fastener types that are not used in this model, like Washers for example, and think of how you would add them into your structure by just adding more classes and not changing anything in the existing structure.
  • Minimize or eliminate attribute re-declaration.
  • Minimize the duplication of code in your hierarchy - choose where the code goes to avoid as much repetition as possible.
  • Every attribute in an instance of one of your Fastener objects must be properly initialized, that is to say each must have a value which cannot be null or zero.
  • You will notice that the legal values for some attributes depend on the object type. And legal values for Finish depend on Material. Otherwise you can assume that fields are independent.
  • Your constructors must throw an exception, called "IllegalFastener" with an appropriate error message if an attempt is made to set any attribute to an illegal value. You must write this exception class.
  • You cannot use protected access - all class methods must be declared private or public and non-final attributes must be private.
  • Remember that methods that do not use attributes can be declared static.
  • Write an accessor method called getOrderCost() in the appropriate class(es) that accepts the number of units of any fastener as an argument and returns the total cost of such an order.
  • You will need toString methods. String representations of fastener objects should contain all field values for that fastener. Here is a sample of what these strings could look like - this is the console output of the JUnit testing program.
  • Write just those methods you need. For example, you cannot write any accessors other than getOrderCost(). All classes are immutable, so you don't need public mutators. Neither do you need compareTo, equals or clone methods.
  • It only makes sense to allow instantiation of your concrete classes - the ones that represent actual fasteners. Figure out a way to prevent the instantiation of your base and intermediate classes.

  Documentation

You do not have to write javadoc comments for your classes. You may use a few normal comments to explain code where needed.

  Submission and Grading Scheme

Please combine all of your *.java files into a single archive file (*.zip or *.rar, for example). Name the file as in "Assn4_NetID.zip" - replacing "NetID" with your ID. Upload this single file into onQ. Do not include your *.class files. Do not include either of the supplied files, your grader will use their own testing classes.

Four marks out of 20 will be allotted to documentation and coding style. Ten marks are allotted for the efficiency and proper design of your hierarchy. The remaining six marks are given for the proper operation and correct results of your code as indicated by passing all the JUnit tests and by the proper operation of the "order system" class.
